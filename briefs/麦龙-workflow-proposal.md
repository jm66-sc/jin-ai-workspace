## 一、理想工作流：麦龙 vs 金龙 分工

### 核心原则

| 维度 | 麦龙（云端） | 金龙（本地） |
|------|------------|------------|
| **擅长** | 分析、决策、写作、多源整合 | 快速执行、本地操作、浏览器自动化 |
| **劣势** | 无法操作本地文件/浏览器 | 模型理解力弱于云端大模型 |
| **时效** | 有定时任务能力，但依赖网络 | 实时，零延迟 |
| **存储** | GitHub / 云端 | 本地磁盘 |

---

### 具体分工

#### 麦龙负责（云端处理）

1. **分析决策层**
   - 解读任务需求，制定执行方案
   - 分析文献、数据、报告
   - 撰写正式文档（方案、报告、邮件）
   - 质量把关（检查金龙输出）

2. **调度协调层**
   - 从 GitHub TASKS.md 领取麦龙的任务
   - 监控外部信息源（网页/公众号/微博 → 摘要）
   - 向微信/钉钉推送结果
   - 更新 TASKS.md 状态

3. **重型计算层**
   - 大规模文献对比分析
   - 多信息源综合摘要
   - 复杂写作任务（个人陈述、方案报告）

#### 金龙负责（本地执行）

1. **本地操作层**
   - 操作浏览器、点击、填表
   - 管理本地文件（下载、分类、整理）
   - 执行需要本地环境的任务（Scrapling 等）

2. **数据采集层**
   - 爬取特定网站（需要登录/验证码）
   - 监控本地硬件状态
   - 文件格式转换（本地软件）

3. **快速响应层**
   - 接收 Boss 临时指令
   - 实时语音/图片处理（本地麦克风/摄像头）

---

### 交接机制

```
Boss 发起任务
    ↓
写入 GitHub TASKS.md（由任意一方完成）
    ↓
金龙/麦龙各自定时扫描 TASKS.md，认领自己的任务
    ↓
执行 → 结果写入 GitHub 指定路径
    ↓
推送微信通知给 Boss
    ↓
Boss 在本地 Mac 定稿 / 反馈
```

**关键：所有任务通过 GitHub 中转，不走各端私有路径。**

## 二、URL 监控方案设计

### 方案定位

> 场景：Boss 给麦龙一批白名单 URL → 麦龙定时爬取 → 摘要 → 推送 GitHub + 微信通知

---

### 工具选型

| 环节 | 工具 | 理由 |
|------|------|------|
| **内容发现** | `batch_web_search` | 关键词监控，发现新页面 |
| **正文提取** | `extract_content_from_websites` | 支持 JS 渲染，能抓动态页面 |
| **摘要生成** | 麦龙大模型自身 | 理解 + 摘要，比任何规则方法都强 |
| **变化检测** | SHA256(url+内容hash) | 简单可靠，有变化才推送 |
| **结果存储** | GitHub intel/{category}/ | 结构化存档，可追溯 |
| **微信通知** | `message` 工具 | 实时推送 |
| **定时调度** | OpenClaw cron | 精确到秒，无需额外服务 |

---

### 监控频率建议

| 场景 | 建议间隔 | 说明 |
|------|---------|------|
| 新闻/资讯类 | 每2小时 | 时效性要求高 |
| 公众号/博客 | 每日2次（9:00 / 21:00） | 中等时效，避免打扰 |
| 竞品/行业报告 | 每日1次 | 低频重要 |
| 官方政策/文件 | 每日1次 | 低频重要 |

---

### 实现流程（伪代码）

```
urls = [白名单URL列表]      # Boss 配置在 GitHub knowledge/url-whitelist.md

每2小时触发:
    for url in urls:
        content = extract_content_from_websites(url)
        hash = sha256(content)
        
        if hash != last_hash[url]:
            summary = 麦龙分析(content, "提取关键信息，写3段摘要")
            
            # 写入 GitHub
            write(f"intel/{category}/{date}/摘要.md", summary)
            
            # 微信通知 Boss
            message: "【监控】{标题} 有更新，已存入 GitHub"
            
            last_hash[url] = hash
        else:
            pass  # 无变化，不打扰
```

---

### 白名单配置格式

```yaml
# knowledge/url-whitelist.md
monitor:
  interval: "2h"
  notify: true
  wechat: true

urls:
  - name: "新能源政策"
    url: "https://www.nea.gov.cn/"
    category: policy
    tags: [能源, 政策]
    
  - name: "锂电池行情"
    url: "https://www.sse.com.cn/"
    category: market
    tags: [锂电, 行情]
```

---

### 注意事项

1. **频率限制**：爬同一域名至少间隔 30 秒，避免被封
2. **robots.txt**：尊重目标站点的 robots.txt
3. **微信去重**：同一 URL 24 小时内不重复通知
4. **错误处理**：超时的 URL 记入错误日志，连续失败3次自动暂停监控
