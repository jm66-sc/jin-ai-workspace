# SmartScout 开发日志

## 重大技术事故记录（2026-02-10）

### 事故概述
**时间**：2026-02-10 22:00-23:00
**阶段**：Phase 1.1 实弹验证
**问题**：Crawl4AI v0.6.3 版本限制导致方法论完全错误
**影响**：抓取质量归零，被迫使用低效Hack方法

---

### 事故详细分析

#### 1. 问题表象
- 抓取结果包含大量页面噪音（导航栏、页头、搜索框）
- 无法获取真实公告列表数据
- 脚本运行效率极低（满屏 `sleep` 和死等）

#### 2. 根本原因
**版本锁定冲突**：
- 项目强制锁定 `crawl4ai==0.6.3`（Python 3.9兼容性要求）
- 但 `v0.6.3` **不支持** `strategy="dynamic"` 参数
- 没有现代智能等待机制，只能手动 `sleep`

**方法论错误**：
- 被迫使用 JavaScript 注入 Hack
- 使用 `querySelectorAll('*')` 暴力遍历（极其业余）
- 无法准确定位列表项选择器

#### 3. 技术对比

**❌ v0.6.3 错误方法**：
```python
# 低效、错误的代码（已废弃）
js_code = """
await new Promise(resolve => setTimeout(resolve, 5000));
window.scrollTo(0, document.body.scrollHeight);
const allElements = document.querySelectorAll('*');  # 暴力遍历！
"""
result = await crawler.arun(url=url, js_code=js_code)
```

**✅ v0.8.x+ 正确方法**：
```python
# 高效、正确的代码（明日升级后使用）
result = await crawler.crawl(
    url=url,
    strategy="dynamic",  # 智能动态渲染
    wait_for=".list-item"  # 精准等待目标元素
)
```

### 事故教训

#### 1. 版本意识
- 不同版本的API可能完全不同
- 必须查阅当前版本的文档，而非假设功能
- 版本兼容性检查应作为技术选型第一步

#### 2. 简单性原则
- 用户说"一行代码搞定"，说明存在简单方法
- 复杂化通常是错误的信号
- 应该先查找最简单API，而不是重新发明轮子

#### 3. 数据验证
- 抓取结果必须验证数据质量，而非仅仅"有数据"
- 页面噪音 ≠ 有效数据
- 必须准确定位目标元素的选择器

## 附录：Crawl4AI版本差异

### v0.6.3（已废弃）
- **Python要求**：3.9+
- **动态渲染**：不支持 `strategy="dynamic"`
- **智能等待**：无内置机制，只能手动 `sleep`
- **推荐度**：❌ 禁止使用

### v0.8.x+（强制升级）
- **Python要求**：3.10+
- **动态渲染**：支持 `strategy="dynamic"` 参数
- **智能等待**：内置 `wait_for` 机制
- **正确用法**：`crawler.crawl(url, strategy="dynamic", wait_for="selector")`
- **推荐度**：✅ 必须使用
